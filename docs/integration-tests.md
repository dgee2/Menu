# Integration Tests

This project uses **.NET Aspire Testing** to run integration tests against the real API, backed by a containerised SQL Server instance and authenticated via Auth0 machine-to-machine tokens.

---

## Architecture Overview

```
┌──────────────────────────────────────────────────┐
│  xUnit Test Runner                               │
│                                                  │
│  ┌──────────────┐   shares    ┌───────────────┐  │
│  │ RecipeTests   │◄──────────►│ ApiTestFixture │  │
│  ├──────────────┤   fixture   │               │  │
│  │ IngredientTests│◄──────────►│  • Builds the │  │
│  ├──────────────┤             │    Aspire host │  │
│  │ RecipeWith    │◄──────────►│  • Starts SQL  │  │
│  │ IngredientsTests│           │    container  │  │
│  └──────────────┘             │  • Runs EF     │  │
│                               │    migrations  │  │
│                               │  • Starts API  │  │
│                               └───────────────┘  │
└──────────────────────────────────────────────────┘
```

### Key Components

| Component | Purpose |
|---|---|
| **`ApiTestFixture`** | Implements `IAsyncLifetime`. Builds a `DistributedApplicationTestingBuilder` from `Menu.AppHost`, starts the Aspire host (SQL Server container, migration service, API), and exposes `GetHttpClient()` which returns an authenticated `HttpClient` pointing at the running API. |
| **`ApiAuthentication`** | Obtains an Auth0 M2M access token using `client_credentials` grant. Credentials are read from user secrets / environment variables. |
| **`TestParameters`** | POCO bound from the `Parameters` configuration section — holds `Auth0TestClientId`, `Auth0TestClientSecret`, `Auth0Audience`, and `Auth0Domain`. |
| **`[Collection("API Host Collection")]`** | All test classes belong to this xUnit collection, backed by `ApiTestFixture` as an `ICollectionFixture`. This ensures a **single Aspire host** is shared across all test classes and that test classes execute **sequentially**. |

### Container Lifetime

`ApiTestFixture` calls `WithContainersLifetime(ContainerLifetime.Session)`, so the SQL Server container persists across the entire test session rather than being recreated per test class. This dramatically speeds up repeated test runs during development.

## Sequential Execution

All integration test classes are decorated with:

```csharp
[Collection("API Host Collection")]
public class MyTests : IClassFixture<ApiTestFixture>
```

xUnit runs all classes within the same `[Collection]` **sequentially** against a single shared `ApiTestFixture` instance. This avoids race conditions when tests create data in the shared database and ensures predictable ordering.

Individual tests within a class also run sequentially (xUnit v2 default for classes sharing a collection fixture).

## Seed Data from Migrations

The EF Core migration `InitialCreate` seeds the following reference data. Tests can assert against these values directly without creating them first.

### UnitType (3 rows)

| Id | Name |
|---|---|
| 1 | Volume |
| 2 | Quantity |
| 3 | Weight |

### Unit (5 rows)

| Id | Name | Abbreviation | UnitTypeId |
|---|---|---|---|
| 1 | Millilitres | ml | 1 (Volume) |
| 2 | Litres | l | 1 (Volume) |
| 3 | Quantity | *(null)* | 2 (Quantity) |
| 4 | Grams | g | 3 (Weight) |
| 5 | Kilograms | kg | 3 (Weight) |

> **No `Ingredient` or `Recipe` rows are seeded.** Tests that need ingredients or recipes must create them via the API first.

---

## Test Data Strategy

- **Reference data** (Units, UnitTypes) — use the seeded values from migrations. Assert against known IDs and names.
- **Ingredients** — create via `POST /api/ingredient` at the start of each test that needs them. Use AutoFixture-generated names to avoid collisions between test runs.
- **Recipes** — create via `POST /api/recipe`. When a recipe needs ingredients, create the required ingredients first via the API.
- **Isolation** — each test creates its own data with unique names (generated by AutoFixture), so tests do not interfere with each other even though they share a database.

## Prerequisites

### 1. Docker

The Aspire host starts a SQL Server container. Docker Desktop (or a compatible container runtime) must be running.

### 2. Auth0 User Secrets

The test project requires Auth0 credentials to obtain access tokens. Set them via user secrets:

```bash
cd MenuApi.Integration.Tests
dotnet user-secrets set "Parameters:Auth0TestClientId" "<your-client-id>"
dotnet user-secrets set "Parameters:Auth0TestClientSecret" "<your-client-secret>"
dotnet user-secrets set "Parameters:Auth0Audience" "<your-audience>"
dotnet user-secrets set "Parameters:Auth0Domain" "<your-domain>"
```

The user secrets ID for the integration test project is `ea3aa4c7-9b32-4485-a5b2-fb1cb0def863`.

### 3. .NET 9 SDK

The solution targets `net9.0`. Ensure the .NET 9 SDK is installed.

## How to Run the Tests

From the solution root:

```bash
dotnet test MenuApi.Integration.Tests
```

Or run a specific test class:

```bash
dotnet test MenuApi.Integration.Tests --filter "FullyQualifiedName~IngredientIntegrationTests"
```

> The first run in a session will be slower as it pulls and starts the SQL Server container. Subsequent runs reuse the persistent container.

## How to Create a New Integration Test

Follow these steps when adding integration tests for a new or existing endpoint.

### 1. Create the Test Class

Add a new `.cs` file in the `MenuApi.Integration.Tests` project root. Apply the collection attribute and inject the fixture:

```csharp
using AwesomeAssertions;
using MenuApi.Integration.Tests.Factory;
using System.Net;
using System.Text;
using System.Text.Json;
using Xunit;

namespace MenuApi.Integration.Tests;

[Collection("API Host Collection")]
public class MyNewEndpointIntegrationTests : IClassFixture<ApiTestFixture>
{
    private readonly JsonSerializerOptions jsonOptions;
    private readonly ApiTestFixture fixture;

    public MyNewEndpointIntegrationTests(ApiTestFixture fixture)
    {
        jsonOptions = new JsonSerializerOptions
        {
            PropertyNamingPolicy = JsonNamingPolicy.CamelCase
        };
        this.fixture = fixture;
    }
}
```

### 2. Define Inner DTOs

Define private/public inner classes that mirror the API response shapes. This avoids coupling integration tests to the API project's view models:

```csharp
private class MyResponse
{
#pragma warning disable S1144
    public int Id { get; set; }
    public string Name { get; set; }
#pragma warning restore S1144
}
```

### 3. Write Test Methods

Use `[Fact]` for tests with no parameters, or `[Theory, ShortStringAutoData]` when AutoFixture should generate test data (e.g. unique names). `ShortStringAutoData` generates strings short enough to fit in database columns (max 20 characters) — standard `AutoData` generates strings that exceed `varchar(50)` limits:

```csharp
[Fact]
public async Task Get_Items_Returns_Ok()
{
    using var client = await fixture.GetHttpClient();
    using var response = await client.GetAsync("/api/my-endpoint");

    await response.ShouldHaveStatusCode(HttpStatusCode.OK);

    var data = await response.Content.ReadAsStringAsync();
    var items = JsonSerializer.Deserialize<List<MyResponse>>(data, jsonOptions);
    items.Should().NotBeNull();
}

[Theory, ShortStringAutoData]
public async Task Create_And_Get_Item(string itemName)
{
    using var client = await fixture.GetHttpClient();

    // POST to create
    var body = new { Name = itemName };
    var content = new StringContent(
        JsonSerializer.Serialize(body, jsonOptions), Encoding.UTF8, "application/json");
    using var postResponse = await client.PostAsync("/api/my-endpoint", content);
    await postResponse.ShouldHaveStatusCode(HttpStatusCode.OK);

    // Parse the created item
    using var stream = await postResponse.Content.ReadAsStreamAsync();
    using var doc = await JsonDocument.ParseAsync(stream);
    var id = doc.RootElement.GetProperty("id").GetInt32();

    // GET to verify
    using var getResponse = await client.GetAsync($"/api/my-endpoint/{id}");
    await getResponse.ShouldHaveStatusCode(HttpStatusCode.OK);
}
```

### 4. Set Up Prerequisite Data via the API

If your endpoint depends on data that isn't seeded by migrations (e.g. an Ingredient must exist before it can be added to a Recipe), create that data via the appropriate `POST` endpoint at the start of your test:

```csharp
[Theory, ShortStringAutoData]
public async Task Create_Recipe_With_Ingredients(string ingredientName, string recipeName)
{
    using var client = await fixture.GetHttpClient();

    // 1. Create the prerequisite ingredient
    var ingredientBody = new { Name = ingredientName, UnitIds = new[] { 4 } };
    var ingredientContent = new StringContent(
        JsonSerializer.Serialize(ingredientBody, jsonOptions), Encoding.UTF8, "application/json");
    using var ingredientResponse = await client.PostAsync("/api/ingredient", ingredientContent);
    await ingredientResponse.ShouldHaveStatusCode(HttpStatusCode.OK);

    // 2. Now create the recipe referencing that ingredient
    var recipeBody = new
    {
        Name = recipeName,
        Ingredients = new[] { new { Name = ingredientName, Unit = "Grams", Amount = 100 } }
    };
    // ... POST and assert
}
```

### 5. Assert Status Codes with `ShouldHaveStatusCode`

Always use the `ShouldHaveStatusCode` extension method (defined in `TestExtensions`) instead of `response.StatusCode.Should().Be(...)`. On failure, it reads and includes the full response body in the assertion message, making it much easier to diagnose 500 errors:

```csharp
await response.ShouldHaveStatusCode(HttpStatusCode.OK);
```

### 6. Extracting Helper Methods

If multiple tests need to create the same type of data, extract a helper method (see `PostIngredientAsync` in `IngredientIntegrationTests` or `PostRecipeAsync` in `RecipeIntegrationTests` for examples):

```csharp
private async Task<(int Id, string Name)> PostItemAsync(HttpClient client, string name)
{
    var body = new { Name = name };
    var content = new StringContent(
        JsonSerializer.Serialize(body, jsonOptions), Encoding.UTF8, "application/json");
    using var response = await client.PostAsync("/api/my-endpoint", content);
    await response.ShouldHaveStatusCode(HttpStatusCode.OK);

    using var stream = await response.Content.ReadAsStreamAsync();
    using var doc = await JsonDocument.ParseAsync(stream);
    return (
        doc.RootElement.GetProperty("id").GetInt32(),
        doc.RootElement.GetProperty("name").GetString()!
    );
}
```

## Logging & Debugging

### API Logs in Test Output

`ApiTestFixture` wires the Aspire host's logging into xUnit via `MartinCostello.Logging.XUnit` using an `IMessageSink`. This means all log output from the API (including EF Core queries, authentication middleware, and unhandled exception details) appears in the test runner's output panel.

The minimum log level is set to `Information` by default. To see more detail (e.g. EF Core SQL queries), change the level in `ApiTestFixture.InitializeAsync`:

```csharp
builder.SetMinimumLevel(LogLevel.Debug);
```

### Response Body on Failure

The `ShouldHaveStatusCode` extension method (in `TestExtensions`) reads the response body when the status code doesn't match expectations and includes it in the assertion failure message. This is invaluable for diagnosing 500 Internal Server Error responses — you'll see the full ProblemDetails JSON or exception message directly in the test output.

### Auth0 Token Caching

The Auth0 M2M token is fetched once and cached for the lifetime of the `ApiTestFixture`. This avoids a round-trip to Auth0 for every test, significantly reducing test execution time.

## API Endpoints Reference

All endpoints are under `/api` and require a valid Bearer token (Auth0).

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/recipe` | List all recipes |
| `POST` | `/api/recipe` | Create a new recipe (with optional ingredients) |
| `GET` | `/api/recipe/{recipeId}` | Get a single recipe with its ingredients |
| `PUT` | `/api/recipe/{recipeId}` | Update a recipe's name and ingredients |
| `GET` | `/api/recipe/{recipeId}/ingredient` | List ingredients for a recipe |
| `GET` | `/api/ingredient` | List all ingredients with their allowed units |
| `GET` | `/api/ingredient/unit` | List all available units |
| `POST` | `/api/ingredient` | Create a new ingredient with associated unit IDs |
